**Source:** https://github.com/Doc-Steve/dendritic-design-with-flake-parts
**Author:** Dr. Steve (special thanks to Victor Borja / @vic)
**Purpose:** A guide and working example for structuring Nix configurations using the Dendritic Pattern with Flake Parts.

---

## What This Repo Is

A documentation guide **plus a working example codebase** for applying the Dendritic Pattern to Nix configurations. The `modules/` tree is real, runnable Nix — not pseudocode. It targets experienced users managing multi-host, multi-platform (NixOS + nix-darwin + Home Manager) configurations.

---

## Core Concepts

### The Dendritic Pattern

A software design pattern — a *way of doing things*, not a library or framework. Two principles:

1. **Feature Modules:** Every `.nix` file (except entry points like `flake.nix`) is a module of the top-level flake-parts configuration. Lower-level NixOS/Home Manager modules are *values defined inside* feature modules, not the feature modules themselves.

2. **Features as building blocks:** Code is organized into self-contained features that group all configuration for a concern across all platforms. Instead of organizing by host, you organize by feature.

### The Configuration Matrix Flip

Traditional (top-down): `myHost → has services, app config, home-manager settings`

Dendritic (bottom-up): `featureX → defines NixOS config + Darwin config + HM config → is imported by hosts`

Adding a feature to a new host is typically one `import` line.

### Aspects

An **aspect** is the `flake.modules.<class>.<name>` representation of a feature for a specific configuration class. One feature file commonly defines multiple aspects — one per class. "Feature" and "aspect" are used interchangeably loosely, but strictly: feature is the concept, aspect is its typed module form.

### Configuration Classes

The classes used in this repo:
- `nixos` — NixOS system modules
- `darwin` — nix-darwin system modules
- `homeManager` — Home Manager user modules
- `generic` — cross-class modules (usable by any class via imports)
- `networkInterface` — custom DRY class for reusable network route blocks

### Flake Parts vs. The Dendritic Pattern

Separate things. Flake Parts (`github:hercules-ci/flake-parts`) is the implementation framework; it provides `flake.modules.<class>.<aspect>` for storing reusable lower-level modules. The pattern itself is tool-agnostic.

## The `flake.nix`

Auto-generated by `vic/flake-file`. The entire output logic is a single line:

```nix
outputs = inputs: inputs.flake-parts.lib.mkFlake { inherit inputs; } (inputs.import-tree ./modules);
```

All inputs are declared here but their URLs are sourced from `flake-file.inputs` declarations scattered across feature modules — you never edit `flake.nix` directly. Regenerate with `nix run .#write-flake`.

Notable input structure:
- `nixpkgs` tracks `nixos-25.11`; `nixpkgs-darwin` tracks `nixpkgs-25.11-darwin` (separate because nix-darwin uses the darwin branch)
- `nix-darwin` follows `nixpkgs-darwin`
- `packages` and `secrets` are local non-flake inputs using `path:`
- `nixpkgs-unstable` is available as `pkgs.unstable` via an overlay defined in `system-minimal`

## The Core Library (`modules/nix/flake-parts []`)

### `dendritic-tools.nix` — Flake-parts bootstrap

Sets up flake-file, import-tree, flake-parts, and supported systems. Every other feature builds on this.

```nix
{ inputs, ... }: {
  flake-file.inputs = {
    flake-parts.url = "github:hercules-ci/flake-parts";
    flake-file.url  = "github:vic/flake-file";
    import-tree.url = "github:vic/import-tree";
  };

  imports = [
    inputs.flake-parts.flakeModules.modules
    inputs.flake-file.flakeModules.default
  ];

  flake-file.outputs = ''
    inputs: inputs.flake-parts.lib.mkFlake { inherit inputs; } (inputs.import-tree ./modules)
  '';

  systems = [ "aarch64-darwin" "aarch64-linux" "x86_64-darwin" "x86_64-linux" ];
}
```

### `lib.nix` — Configuration builder helpers

Defines `flake.lib` with three builder functions used by host `flake-parts.nix` files:

```nix
{ inputs, lib, ... }: {
  options.flake.lib = lib.mkOption {
    type = lib.types.attrsOf lib.types.unspecified;
    default = {};
  };

  config.flake.lib = {
    mkNixos = system: name: {
      ${name} = inputs.nixpkgs.lib.nixosSystem {
        modules = [
          inputs.self.modules.nixos.${name}
          { nixpkgs.hostPlatform = lib.mkDefault system; }
        ];
      };
    };

    mkDarwin = system: name: {
      ${name} = inputs.nix-darwin.lib.darwinSystem {
        modules = [
          inputs.self.modules.darwin.${name}
          { nixpkgs.hostPlatform = lib.mkDefault system; }
        ];
      };
    };

    mkHomeManager = system: name: {
      ${name} = inputs.home-manager.lib.homeManagerConfiguration {
        pkgs = inputs.nixpkgs.legacyPackages.${system};
        modules = [
          inputs.self.modules.homeManager.${name}
          { nixpkgs.config.allowUnfree = true; }
        ];
      };
    };
  };
}
```

These are called from host and user `flake-parts.nix` files:
```nix
flake.nixosConfigurations    = inputs.self.lib.mkNixos       "x86_64-linux"   "homeserver";
flake.darwinConfigurations   = inputs.self.lib.mkDarwin      "aarch64-darwin" "macbook";
flake.homeConfigurations     = inputs.self.lib.mkHomeManager "x86_64-linux"   "bob";
```

### `factory.nix` — Factory function registry

```nix
{ lib, ... }: {
  options.flake.factory = lib.mkOption {
    type = lib.types.attrsOf lib.types.unspecified;
    default = {};
  };
}
```

Factory functions are stored as `config.flake.factory.<name>` and accessed via `self.factory.<name>` or `inputs.self.factory.<name>` in feature modules.

### `darwinConfigurations-fix.nix`

Manually adds `flake.darwinConfigurations` as a flake-parts option since no official nix-darwin flake-parts module exists yet:

```nix
{ lib, flake-parts-lib, ... }: {
  options.flake = flake-parts-lib.mkSubmoduleOptions {
    darwinConfigurations = lib.mkOption {
      type = lib.types.lazyAttrsOf lib.types.raw;
      default = {};
    };
  };
}
```

## System Type Hierarchy (Inheritance Aspect Chain)

The four system types form a layered inheritance chain. Each level imports the previous and adds more features. All four define aspects for `nixos`, `darwin`, and `homeManager` simultaneously in a single file (except level 1 which is split by class into subdirectories).

### Level 1: `system-minimal` — Base Nix settings

Level 1 is split into subdirectories per class because the per-class content is substantial and includes separate `flake-parts.nix` input declarations.

**NixOS** (`1 - system-minimal/nixos/nixos-minimal.nix`):
```nix
{ inputs, ... }: {
  flake.modules.nixos.system-minimal = { pkgs, ... }: {
    nixpkgs.overlays = [(final: _prev: {
      unstable = import inputs.nixpkgs-unstable {
        inherit (final) config;
        system = pkgs.stdenv.hostPlatform.system;
      };
    })];
    nixpkgs.config.allowUnfree = true;
    system.stateVersion = "25.05";
    nix.settings = {
      substituters = [ "https://cache.nixos.org?priority=10" "https://nix-community.cachix.org" ... ];
      experimental-features = [ "nix-command" "flakes" ];
      trusted-users = [ "root" "@wheel" ];
      download-buffer-size = 1024 * 1024 * 1024;
    };
    nix.extraOptions = "warn-dirty = false\nkeep-outputs = true";
  };
}
```

**Darwin** (`1 - system-minimal/darwin/darwin-minimal.nix`) uses `determinateNix.customSettings` instead of `nix.settings` (because Determinate Nix manages the daemon). Adds the `unstable` overlay and darwin utility packages (`darwin-option`, `darwin-rebuild`, etc). `macos-apps-fix.nix` adds a Home Manager activation script that copies `.app` bundles into `~/Applications/Nix Apps/` so Spotlight can find them.

**Home Manager** (`1 - system-minimal/homeManager/homeManager-minimal.nix`):
```nix
{ inputs, ... }: {
  flake.modules.homeManager.system-minimal = { config, pkgs, lib, ... }: {
    home.homeDirectory = if pkgs.stdenv.isDarwin
      then (lib.mkForce "/Users/${config.home.username}")
      else "/home/${config.home.username}";
    home.stateVersion = "23.05";
  };
}
```

### Level 2: `system-default` — Adds essential Nix tools

```nix
{ inputs, ... }: {
  flake.modules.nixos.system-default = {
    imports =
      with inputs.self.modules.nixos;   [ system-minimal home-manager secrets ]
      ++ with inputs.self.modules.generic; [ systemConstants pkgs-by-name ];
  };

  flake.modules.darwin.system-default = {
    imports =
      with inputs.self.modules.darwin;  [ system-minimal determinate home-manager homebrew secrets ]
      ++ with inputs.self.modules.generic; [ systemConstants pkgs-by-name ];
  };

  # impermanence is NOT added here — no Darwin implementation, must be added per-host manually
  flake.modules.homeManager.system-default = {
    imports =
      with inputs.self.modules.homeManager; [ system-minimal secrets ]
      ++ [ inputs.self.modules.generic.systemConstants ];
  };
}
```

### Level 3: `system-cli` — Adds SSH, firmware, CLI tools, shell

```nix
{ inputs, ... }: {
  flake.modules.nixos.system-cli = {
    imports = with inputs.self.modules.nixos; [ system-default ssh firmware cli-tools ];
  };
  flake.modules.darwin.system-cli = {
    imports = with inputs.self.modules.darwin; [ system-default ssh cli-tools ];
  };
  flake.modules.homeManager.system-cli = {
    imports = with inputs.self.modules.homeManager; [ system-default shell ];
  };
}
```

### Level 4: `system-desktop` — Adds printing, browser, office

```nix
{ inputs, ... }: {
  flake.modules.nixos.system-desktop = {
    imports = with inputs.self.modules.nixos; [ system-cli printing ];
  };
  flake.modules.darwin.system-desktop = {
    imports = with inputs.self.modules.darwin; [ system-cli ];
  };
  flake.modules.homeManager.system-desktop = {
    imports = with inputs.self.modules.homeManager; [ system-cli browser office ];
  };
}
```

## Hosts

### `homeserver` (NixOS, x86_64-linux)

The homeserver feature is split across many files and subdirectories — all contribute to `flake.modules.nixos.homeserver` and are merged by the module system.

**`flake-parts.nix`** — registers the flake output:
```nix
{ inputs, ... }: {
  flake.nixosConfigurations = inputs.self.lib.mkNixos "x86_64-linux" "homeserver";
}
```

**`configuration.nix`** — core imports:
```nix
{ inputs, ... }: {
  flake.modules.nixos.homeserver = {
    imports = with inputs.self.modules.nixos; [ system-cli systemd-boot impermanence ];
  };
}
```

**`filesystem.nix`** — uses the Constants Aspect to read shared values:
```nix
{ flake.modules.nixos.homeserver = { config, ... }: {
  fileSystems."/" = { device = "/dev/sda"; };
  services.zfs.zed.settings = {
    ZED_EMAIL_ADDR = [ config.systemConstants.adminEmail ]; # ← constant from generic class
  };
}; }
```

**`network.nix`** — uses DRY Aspects merged with host-specific addresses:
```nix
{ self, lib, ... }: {
  flake.modules.nixos.homeserver = {
    networking.interfaces."enp86s0" = with self.modules.networkInterface;
      lib.mkMerge [
        subnet-A
        subnet-B
        {
          ipv4.addresses = [{ address = "10.0.0.1"; prefixLength = 16; }];
          ipv6.addresses = [{ address = "2001:1470:fffd:2098::e006"; prefixLength = 64; }];
        }
      ];
  };
}
```

**`services/syncthing/syncthing.nix`** — imports syncthing feature and adds host-specific folder config:
```nix
{ inputs, ... }: {
  flake.modules.nixos.homeserver = {
    imports = with inputs.self.modules.nixos; [ syncthing ];
    services.syncthing.settings.folders."sharedfolder" = {
      enable = true; path = "/sharedfolder"; type = "sendreceive";
    };
  };
}
```

**`services/syncthing/syncthingDevices.nix`** — Collector Aspect: contributes to the `syncthing` aspect from within the homeserver directory:
```nix
{
  # Note: contributes to flake.modules.nixos.syncthing, NOT homeserver
  flake.modules.nixos.syncthing = {
    services.syncthing.settings.devices.homeserver = {
      id = "VNV2XTI-6VY6KR2-OCASMST-Z35JUEG-VNV2XTI-KJWBOKQ-6VYUKR2-Z35JUEG";
    };
  };
}
```

**`users/mallory.nix`** and **`users/eve.nix`** — add users to the host:
```nix
{ inputs, ... }: {
  flake.modules.nixos.homeserver = {
    imports = with inputs.self.modules.nixos; [ mallory ];
  };
}
```

## Users

### `bob` — Multi-platform (NixOS + Darwin + standalone HM)

`bob.nix` demonstrates using `lib.mkMerge` to combine Factory Aspect output with per-class customizations. This is the key pattern for the user Factory Aspect.

```nix
# modules/users/bob [NDn]/bob.nix
{ self, lib, ... }: {
  flake.modules = lib.mkMerge [
    (self.factory.user "bob" true)    # ← Factory generates nixos.bob, darwin.bob, homeManager.bob
    {
      nixos.bob = {
        imports = with self.modules.nixos; [ /* developmentEnvironment */ ];
        users.users.bob.group = "audio";   # ← extend factory-generated user config
      };
      darwin.bob = {
        imports = with self.modules.darwin; [ /* drawingApps */ ];
      };
      homeManager.bob = { pkgs, ... }: {
        imports = with self.modules.homeManager; [
          system-desktop
          # adminTools vscode passwordManager
        ];
        home.packages = with pkgs; [ mediainfo ];
      };
    }
  ];
}
```

`flake-parts.nix` gives bob a standalone Home Manager configuration (in addition to being embedded in hosts):
```nix
{ inputs, ... }: {
  flake.homeConfigurations = inputs.self.lib.mkHomeManager "x86_64-linux" "bob";
}
```

## Factory Aspects (Real Code)

### `factory/user [ND]` — User factory

Takes `username` and `isAdmin` and returns an attribute set with `nixos`, `darwin`, and `homeManager` module-valued keys — suitable for merging directly into `flake.modules`:

```nix
{ self, ... }: {
  config.flake.factory.user = username: isAdmin: {

    nixos."${username}" = { lib, pkgs, ... }: {
      users.users."${username}" = {
        isNormalUser = true;
        home = "/home/${username}";
        extraGroups = lib.optionals isAdmin [ "wheel" ];
        shell = pkgs.zsh;
      };
      programs.zsh.enable = true;
      home-manager.users."${username}".imports = [ self.modules.homeManager."${username}" ];
    };

    darwin."${username}" = { lib, pkgs, ... }: {
      users.users."${username}" = { home = "/Users/${username}"; shell = pkgs.zsh; };
      home-manager.users."${username}".imports = [ self.modules.homeManager."${username}" ];
      system.primaryUser = lib.mkIf isAdmin username;
      programs.zsh.enable = true;
    };

    homeManager."${username}" = {
      home.username = username;
    };
  };
}
```

Usage in `bob.nix`:
```nix
flake.modules = lib.mkMerge [
  (self.factory.user "bob" true)
  { nixos.bob = { users.users.bob.group = "audio"; }; }
];
```

### `factory/mount-cifs-nixos [N]` — CIFS mount factory

Returns an anonymous NixOS module (no named aspect — the result goes directly into `imports`):

```nix
{ config.flake.factory.mount-cifs-nixos =
  { host, resource, destination, credentialspath, UID, GID }:
  { config, lib, ... }: {
    fileSystems."${destination}" = {
      device = "//${host}/${resource}";
      fsType = "cifs";
      options =
        let
          automount-opts = [ "x-systemd.automount" "noauto" "nofail" "soft"
            "x-systemd.idle-timeout=60" "x-systemd.device-timeout=5s" "x-systemd.mount-timeout=5s" ];
          mount-opts = [ "rw" "iocharset=utf8" ];
          user = [ "uid=${UID}" "gid=${GID}" ];
          credentials = [ "credentials=${credentialspath}" ];
        in
        automount-opts ++ mount-opts ++ user ++ credentials;
    };
  };
}
```

## Aspect Pattern Examples (Real Code)

### Simple Aspect — `services/ssh [ND]`

Independent aspects per OS class in a single file:

```nix
{
  flake.modules.nixos.ssh = {
    services.openssh = {
      enable = true; openFirewall = true;
      settings = { PasswordAuthentication = true; PermitRootLogin = "yes"; };
    };
  };
  flake.modules.darwin.ssh = {
    services.openssh.enable = true;
  };
}
```

### Simple Aspect Split Across Files — `programs/cli-tools [ND]`

Three files all contribute to the same `nixos.cli-tools` and `darwin.cli-tools` aspects. The module system merges them automatically.

```nix
# generic.nix — shared packages via an inline anonymous module
let
  genericPackages = { pkgs, ... }: {
    environment.systemPackages = with pkgs; [ git tmux home-manager local.cowsay ];
  };
in {
  flake.modules.nixos.cli-tools  = { imports = [ genericPackages ]; };
  flake.modules.darwin.cli-tools = { imports = [ genericPackages ]; };
}

# nixos.nix — NixOS-only additions
{ flake.modules.nixos.cli-tools = { pkgs, lib, ... }: {
  environment.systemPackages = with pkgs; [ parted ]
    ++ lib.optionals (pkgs.stdenv.hostPlatform.system == "x86_64-linux") [ intel-gpu-tools ];
}; }

# darwin.nix — Darwin-only additions
{ flake.modules.darwin.cli-tools = { pkgs, ... }: {
  environment.systemPackages = with pkgs; [ mas ];
}; }
```

### Multi Context Aspect — `programs/gnome [N]`

NixOS aspect embeds a Home Manager auxiliary aspect via `home-manager.sharedModules`. The HM aspect is also independently usable.

```nix
# gnome.nix
{ inputs, ... }: {
  flake.modules.nixos.gnome = {
    home-manager.sharedModules = [ inputs.self.modules.homeManager.gnome ]; # ← embed HM aux
    services.xserver.displayManager.gdm.enable = true;
    services.xserver.desktopManager.gnome.enable = true;
    programs.dconf.enable = true;
  };

  flake.modules.homeManager.gnome = {   # ← auxiliary HM aspect (also usable standalone)
    dconf.settings."org/gnome/desktop/interface".enable-hot-corners = true;
  };
}

# extensions.nix — additional file contributing to the same HM aspect
{ flake.modules.homeManager.gnome = { pkgs, ... }: {
  home.packages = with pkgs; [ gnome-extension-manager gnomeExtensions.caffeine ];
  dconf.settings."org/gnome/shell" = {
    disable-user-extensions = false;
    enabled-extensions = [ "caffeine@patapon.info" ];
  };
}; }
```

### Conditional Aspect — `programs/office [nd]`

Uses `lib.mkMerge` with `lib.mkIf` for platform-conditional packages:

```nix
{ flake.modules.homeManager.office = { pkgs, lib, config, ... }:
  lib.mkMerge [
    {
      home.packages = with pkgs; [ pdfarranger notesnook ];
    }
    (lib.mkIf pkgs.stdenv.isLinux {
      home.packages = with pkgs; [ libreoffice-qt6 gimp3-with-plugins ];
    })
    (lib.mkIf pkgs.stdenv.isDarwin {
      home.packages = with pkgs; [ libreoffice-bin brewCasks.gimp ]; # ← brew-nix cask
    })
  ];
}
```

### Collector Aspect — `services/syncthing [N]`

The base `syncthing` aspect lives in `services/`. Each host contributes its Syncthing device ID from within its own directory, without the syncthing feature needing to know about specific hosts.

```nix
# services/syncthing [N]/syncthing.nix — base definition
{ flake.modules.nixos.syncthing = {
  services.syncthing = { enable = true; openDefaultPorts = true; };
}; }

# hosts/homeserver [N]/services/syncthing/syncthingDevices.nix — contribution
{ flake.modules.nixos.syncthing = {   # ← same aspect, merged automatically
  services.syncthing.settings.devices.homeserver = {
    id = "VNV2XTI-6VY6KR2-OCASMST-Z35JUEG-VNV2XTI-KJWBOKQ-6VYUKR2-Z35JUEG";
  };
}; }
```

### Constants Aspect — `system/settings/systemConstants [NDnd]`

Uses the `generic` class to make constants available across all other classes:

```nix
{ flake.modules.generic.systemConstants = { lib, ... }: {
  options.systemConstants = lib.mkOption {
    type = lib.types.attrsOf lib.types.unspecified;
    default = {};
  };
  config.systemConstants = {
    adminEmail = "admin@test.org";
  };
}; }
```

Imported in `system-default` for all classes. Consumed in `homeserver/filesystem.nix`:

```nix
services.zfs.zed.settings.ZED_EMAIL_ADDR = [ config.systemConstants.adminEmail ];
```

### DRY Aspect — `system/settings/network`

Uses a custom `networkInterface` class to define reusable route blocks:

```nix
# subnet-A.nix
{ flake.modules.networkInterface.subnet-A = {
  ipv6.routes = [{ address = "2001:1470:fffd:2098::"; prefixLength = 64; via = "fdfd:b3f0::1"; }];
```

```nix
    subnet-A
    subnet-B
    { ipv4.addresses = [{ address = "10.0.0.1"; prefixLength = 16; }]; }
```

## Nix Tools Integration

Each tool in `modules/nix/tools/` follows the same pattern: a `flake-parts.nix` that declares inputs and imports at the flake level, and one or more `.nix` files defining the aspects.

### `impermanence [N]`

Also defines a `flake.lib` helper so features can guard their persistence config safely:

```nix
# impermanence.nix
{ inputs, ... }: {
  flake.lib = {
    mkIfPersistence = config: settings:
      if config ? home
      then (if config.home ? persistence then settings else {})
      else (if config.environment ? persistence then settings else {});
  };

  flake.modules.nixos.impermanence = {
    imports = [ inputs.impermanence.nixosModules.impermanence ];
  };
}

# minimum.nix — base persistence directories added to the impermanence aspect
{ flake.modules.nixos.impermanence = { config, ... }: {
  environment.persistence."/persistent" = {
    hideMounts = true;
    directories = [ "/var/log" "/var/lib/nixos" "/var/lib/systemd/coredump"
                    "/etc/NetworkManager/system-connections" ];
    files = [ "/etc/machine-id" ];
  };
  home-manager.sharedModules = [{ home.persistence."/persistent" = {}; }];
  programs.fuse.userAllowOther = true;
}; }
```

Features use `mkIfPersistence` to be safe on hosts both with and without impermanence:

```nix
# programs/browser [nd]/impermanence.nix
{ inputs, ... }: {
  flake.modules.homeManager.browser = { config, lib, pkgs, ... }: {
    home = inputs.self.lib.mkIfPersistence config {
      persistence."/persistent".directories =
        [ ".mozilla/firefox" ]
        ++ lib.optionals (lib.elem pkgs.google-chrome config.home.packages) [ ".config/Google-Chrome" ]
        ++ lib.optionals (lib.elem pkgs.chromium config.home.packages) [ ".config/chromium" ];
    };
  };
}
```

### `secrets [NDnd]` — agenix for all classes

Single file covers all four classes:

```nix
{ inputs, self, ... }: {
  flake.modules.nixos.secrets    = { pkgs, ... }: { imports = [ inputs.agenix.nixosModules.default ]; ... };
  flake.modules.darwin.secrets   = { pkgs, ... }: { imports = [ inputs.agenix.darwinModules.default ]; ... };
  flake.modules.homeManager.secrets = { pkgs, ... }: { imports = [ inputs.agenix.homeManagerModules.default ]; };

```

### `home-manager [ND]`

`flake-parts.nix` imports the HM flake module (makes `home-manager.*` options available at the system level). `home-manager.nix` defines shared config and creates NixOS + Darwin aspects:

```nix
let home-manager-config = { lib, ... }: {
  home-manager = {
    verbose = true; useUserPackages = true; useGlobalPkgs = true;
    backupFileExtension = "backup"; backupCommand = "rm"; overwriteBackup = true;
  };
}; in {
  flake.modules.nixos.home-manager = {
    imports = [ inputs.home-manager.nixosModules.home-manager home-manager-config ];
  };
  flake.modules.darwin.home-manager = {
    imports = [ inputs.home-manager.darwinModules.home-manager home-manager-config ];
  };
}
```

### `pkgs-by-name [G]` — local packages overlay

Makes `packages/` available as `pkgs.local.*` in all systems. `packages/cowsay/package.nix` wraps the standard cowsay to always be "sleepy" (`--add-flag "-s"`). Referenced as `local.cowsay` in `cli-tools`.

### `determinate [D]`

Darwin uses Determinate Nix for daemon management. The aspect disables the standard nix module and defers to Determinate:

```nix
{ inputs, ... }: {
  flake.modules.darwin.determinate = {
    imports = [ inputs.determinate.darwinModules.default ];
    nix.enable = false; # Determinate Nix handles the Nix configuration
  };
}
```

## Key Rules Observed in the Real Code

**`flake-parts.nix` files are always separate.** Any file that produces a flake output (`nixosConfigurations`, `darwinConfigurations`, `homeConfigurations`) or declares `flake-file.inputs` uses a dedicated `flake-parts.nix` alongside the feature's regular `.nix` files.

**Hosts never define user accounts directly.** User system settings live in `users/<name>/<name>.nix`. The host's `users/<name>.nix` is just an import of the user aspect plus host-specific overrides (e.g., which secrets to decrypt, home-manager user passthrough).

**`lib.mkMerge` is mandatory for combining attribute sets** — never `//`. This applies when combining Factory Aspect output with customizations, when building Conditional Aspects, and when merging DRY aspects for things like network interfaces.

**`self` vs `inputs.self`.** Both appear in the code. `self` works as a module argument directly. `inputs.self` is used when `self` is not in scope (e.g., inside a `let` binding's returned value, or when only `inputs` is available).

**Impermanence is opt-in per host.** It's imported in `homeserver/configuration.nix` but not for `linux-desktop` or `macbook`. Features with persistence config use `mkIfPersistence` to remain safe on either type of host.

**Darwin uses Determinate Nix** for managing the Nix daemon — `nix.settings` is replaced by `determinateNix.customSettings` in `darwin-minimal.nix`.

**The `generic` class** is the cross-class sharing mechanism. `systemConstants` and `pkgs-by-name` are `generic` aspects imported by `system-default` into every class.

**Splitting a feature across files is normal and encouraged.** All files under `modules/` are auto-imported by `import-tree`; the module system merges all contributions to the same aspect automatically. The `homeserver` host is a good example: hardware, network, filesystem, services, and users are all separate files.

**Files with `/_` anywhere in their path are ignored by `import-tree`** — use this to disable WIP modules without deleting them.

## Notes for AI Assistant

- `specialArgs` is an anti-pattern in dendritic configs. Redirect users to `let` bindings or custom flake-parts options instead.
- Conditional imports (`lib.mkIf` inside `imports = [...]`) are forbidden — make module *content* conditional, not the import list.
- Always use `lib.mkMerge` when combining attribute sets in module code, never `//`.
- `flake.lib` (defined in `lib.nix`) holds `mkNixos`, `mkDarwin`, `mkHomeManager`, and `mkIfPersistence`. Access as `inputs.self.lib.*` or `self.lib.*`.
- `flake.factory` (defined in `factory.nix`) holds factory functions. Access as `inputs.self.factory.*` or `self.factory.*`.
- Host `flake-parts.nix` files are the only place that calls `mkNixos`/`mkDarwin`/`mkHomeManager`.
- Darwin has no official flake-parts module — `darwinConfigurations-fix.nix` manually declares the option.
- `flake.nix` is auto-generated by `vic/flake-file`. Declare inputs via `flake-file.inputs` in feature modules; regenerate with `nix run .#write-flake`.
- The 8 Aspect patterns: Simple, Multi Context, Inheritance, Conditional, Collector, Constants, DRY, Factory. All are demonstrated in the example repo with real code.
- `import-tree` ignores paths containing `/_`.
